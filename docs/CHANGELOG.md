### Behavior or Interface Changes
- **CRITICAL RUID-policy fix.** `download_submission_images.py` now resolves every form row through `ruid_resolver.resolve_form_row_to_roster_row` before any image is saved, so every filename uses the authoritative **Roster RUID** (resolved against `Protein_Images/semesters/<term>/roster.csv`), not the **Form RUID** typed by the student. Before this change, the downloader used the raw cell that started with `'900'` / `'960'` verbatim, so files like `900593554-protein01-pa1-meet_gulabani.png` were saved under the typed-but-wrong RUID; the example reported by the operator should have been `900593544`. The fix applies uniformly to all four locations: `<image_dir>/raw/`, `<image_dir>/trim/`, `<image_bank>/<term>/<image_dir>/raw/`, `<image_bank>/<term>/<image_dir>/trim/`. Rows the matcher cannot resolve are quarantined: no image is saved, the row is appended to `<csv_dir>/quarantine.log` with the top roster candidates the matcher considered, and the HTML profile page shows a red `QUARANTINED` block so the operator sees the gap. The shared matcher and `assigned_ruids` set are constructed once in `main()` and passed through `process_one_csv -> generate_html`, so a Form RUID resolved in one CSV cannot silently re-resolve to the same roster row in a later CSV during a `--all` run. Form CSV column names are hardcoded as module-level constants (`COL_STUDENT_ID`, `COL_FIRST_NAME`, `COL_LAST_NAME`, `COL_USERNAME`) per `docs/PYTHON_STYLE.md` argparse minimalism; the Google Form headers are fixed per semester, so per-run flags would be over-engineering.
- **Grader migration.** `student_id_protein.match_lists_and_add_student_ids` now also calls `ruid_resolver.resolve_form_row_to_roster_row` instead of an inline `RosterMatcher.match` loop. Same grader-side behavior preserved (interactive matcher, `auto_threshold=0.90`, raise on miss or duplicate); just routed through the shared single function so both download and grade have one place to fix bugs or change policy.
- `download_submission_images.py` now uses `archive_paths.make_archive_assignment_dir(term, image_dir.name)` to build the archive root instead of constructing it inline. The previous inline build was wrapped in a broad `except Exception` that silently disabled archiving on any failure; this is replaced with a direct call that surfaces real errors.
- `local_migrations/migrate_image_bank_to_terms.py` now removes the source file when MD5 matches the destination ("identical-md5 -> delete source" per plan). Without this, a partial first run would leave source files in place and `os.rmdir` could not clean the source folder. Re-runs are now truly idempotent.

### Decisions and Failures
- Existing wrong-RUID files already on disk under `<image_dir>/raw/`, `<image_dir>/trim/`, and the bank mirrors are **not** renamed by the RUID-policy fix. Operator must clear semester `raw/`+`trim/` and re-run `download_submission_images.py` to regenerate canonical filenames. The bank still holds the typed-RUID copies; a one-shot rename helper is left as a follow-up.
- An earlier draft of this change introduced `protein_image_grader/ruid_aliases.py` to persist a `typed_ruid -> resolved_ruid` cache as `Protein_Images/semesters/<term>/ruid_aliases.yml`. Removed before merge. Reasoning: `roster.csv` is the only source of truth; a typed RUID that misses the roster is a one-off student typo, not durable data worth caching across runs. The matcher re-runs on every download (cheap), and `quarantine.log` already surfaces the typos for triage. A persistent alias cache would create a second source of truth that could disagree with the roster.
- An earlier draft of `ruid_resolver.py` exposed a class-heavy `RuidResolver` with `load`/`flush`/`resolve` lifecycle. Reduced before merge to one module-level function (`resolve_form_row_to_roster_row`) plus the two result dataclasses. Reasoning: with the alias cache gone, the class held no real state worth packaging; both consumers can build the matcher and the `assigned_ruids` set themselves, then call one function per row.
- Image bank lifecycle policy is now codified in `docs/IMAGE_BANK_POLICY.md`: dual-write on download, no script ever auto-deletes either copy. Operator-only manual cleanup at term end. See doc for the verification that dual-write does not break duplicate detection.

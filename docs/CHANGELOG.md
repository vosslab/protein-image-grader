## 2026-05-10

### Behavior or Interface Changes
- `protein_image_grader/start_grading.py` `render_footer_warnings` is silent when the chosen checkpoint is `output-protein_image_NN.yml`. Shallower `*_save.yml` files left on disk after a successful run are normal scaffolding, not stale state, and listing them every dashboard refresh was just noise. The "image NN: using <label> checkpoint <chosen>; also found: <comma-list>" advisory now fires only when the chosen checkpoint is shallower than `output` (i.e. a crashed or in-flight run) so a real signal still surfaces.
- `protein_image_grader/start_grading.py` regrade prompt simplified. The PARTIAL-auto-route prompt no longer claims `any hand-edits to <csv> will not be preserved` -- that line was misleading because the export CSV is regenerated from the YAML on every grade run and is never hand-edited. Prompt is now `Re-grade to pick up N new row(s)?`.
- `protein_image_grader/start_grading.py` `build_download_command` now passes `--trim --rotate` so orchestrator-launched downloads always produce both the raw image and the trimmed/rotated companion. Direct invocations of `download_submission_images.py` still default to off; this is a belt-and-suspenders default for users who do not run the manual `-rat` step first.
- `protein_image_grader/start_grading.py` email step gains a TTY-only promote-to-real prompt. After the dry-run subprocess exits 0, `run_step` calls `confirm_overwrite("Dry-run complete. Send for real now?", default_yes=False)`; on Y it re-runs the same command with `-e` appended. Direct `send_feedback_email.py` calls do NOT prompt -- they keep the existing `-e`/`--dry-run` dichotomy.
- `protein_image_grader/send_feedback_email.py` dry-run is now end-to-end instead of just a log write. For every student: build `make_content` body, build `make_email_header` recipient list, compose the AppleScript via `applescript_dispatch.compose_script`, and print the AppleScript so a malformed body or recipient surfaces here, not during the real send. For the FIRST non-skipped student only: dispatch the AppleScript to a new module-level constant `DRY_RUN_PREVIEW_ADDRESS = "nvoss@roosevelt.edu"` (instructor domain, not student domain) via `applescript_dispatch.run_script`. Subject is prefixed with `[DRY-RUN PREVIEW]`. An exception from `run_script` propagates uncaught and halts the loop, which is preferred over logging hundreds of `dry_run` rows after the dispatcher is known broken. The legacy dry-run only printed a one-liner per student and was insufficient -- this matches the pre-canonical behavior in commit 53a23dd, which composed the AppleScript and dispatched a single test send to the instructor.
- `protein_image_grader/send_feedback_email.py` dual-send to non-roosevelt domains. After `FIX_DOMAINS` correction, any typed email whose domain is not `mail.roosevelt.edu` is added as a second recipient alongside the derived `<username>@mail.roosevelt.edu`. Previously, `ACCEPTABLE_DOMAINS = ['gmail.com', 'mail.roosevelt.edu']` gated the dual send and any other domain (yahoo, hotmail, outlook, icloud, etc.) was silently dropped with only a stderr warning. `ACCEPTABLE_DOMAINS` is removed.
- `protein_image_grader/send_feedback_email.py` `_student_typed_email(student_entry)` helper probes the historical column-name aliases `email`, `Email Address`, `Email` for the student-typed email address and returns `""` when none is present. Both call sites (`make_email_header`, `make_content`) skip the typed-vs-derived sanity check when no typed email is on file, fixing a `KeyError: 'email'` regression that crashed `send_feedback_email.py` on every YAML row that did not carry a lowercase `email` key (i.e. all of them in the current schema).
- `protein_image_grader/send_feedback_email.py` `make_email_header` warns and continues when the typed email is malformed (no `@` sign). Previously the run aborted with `ValueError: not enough values to unpack`.
- `protein_image_grader/send_feedback_email.py` FIX_DOMAINS auto-correction now prints `email typo for <name>: <typed-domain> -> <corrected-domain>` per row so the operator can fix the form CSV in addition to the per-run silent rewrite. An expanded provider-typo map was tried and reverted; new entries land when real errors show up in the wild.

The legacy `profiles_*` name dated to a previous student-profile downloader and was misleading here. `protein_image_grader/grade_protein_image.py` `--make-html` path also moved from the legacy `profiles.html` to per-image `protein_images_NN.html`. ### Decisions and Failures
- YAML is the single source of truth for graded records; the output CSV is downstream and is no longer read by the dashboard. The earlier "CSV is graded-state truth" model meant a grader run that wrote the YAML but failed before the CSV write would silently mis-report PARTIAL. Migrating the dashboard to read the YAML closes that gap and matches the operator's intuition.
- Student identity is `Student ID` only. Resubmissions (same student, new timestamp) are NOT auto-regraded; the cached YAML row wins. To regrade a specific student the operator edits the YAML per the Manual operator contract (set `Image Assessment Complete: false` to re-prompt image questions, delete a `<Q> Status` key to re-prompt one CSV question, delete the entire student row to fully regrade). An earlier draft used `(Student ID, timestamp)` as the merge key to mirror `csv_compare`'s record-vs-record contract; rejected before implementation because (a) it disagreed with the grader's "one student, one graded record" emission contract and (b) it would have made the manual escape hatch (delete the row) unintuitive.
- Dashboard is non-interactive. Multiple checkpoints on disk surface as a footer warning naming the chosen file and listing the others; only parse / validation failure flips status to CONFLICT. An earlier draft proposed prompting the operator at the dashboard when multiple checkpoints existed and again at the regrade entry on conflict; rejected because the dashboard is the status surface and prompts there break automation. CONFLICT + a clear footer reason is the substitute.
- Mtime-based "stale-deepest" rule rejected. An earlier draft flagged conflict when a shallower checkpoint had a newer mtime than the deepest; dropped because operators may legitimately touch files (e.g., to force a re-stat). The picker trusts the catalog precedence, not mtimes.
- No new CLI flag. An earlier draft introduced `--choose-checkpoint <path>` for the rare same-rank tie; rejected because the existing `--yaml-backup-file` already accepts an explicit override and the per-PYTHON_STYLE.md ARGPARSE rule forbids adding flags users do not need to change between runs. The orchestrator picks; the grader honors the path.
- Manual operator contract is limited to behaviors that are tested. The plan called out three levers (`Image Assessment Complete: false`, delete `<Q> Status`, delete the YAML row) that already work because the existing pipeline reads those exact fields as skip gates. Hash- and download-forcing claims (e.g., "clear hash fields and the next run re-downloads") are intentionally out of scope until the existing `read_save_images` cache path is verified end-to-end and a test is added; the supported lever is "clear `Image Format`" because that is the field the cache check actually keys on.
- Duplicate detection refactor declined as out of scope. Cached YAML rows already carry their md5 + perceptual hashes, so `fill_local_image_hashes` produces identical output whether a row was cached or freshly computed; rerunning duplicate detection on the merged tree is correct without changes.

### Decisions and Failures
- Strict superset for the form-CSV re-import collision check is *keyed by `(Student ID, timestamp)`* rather than a byte-identical row prefix. An earlier draft of the plan used prefix-equal matching; rejected before implementation because Google Forms re-exports occasionally re-order rows and a prefix check would (wrongly) raise on a real "no data changed, just re-exported" CSV. The keyed comparator still raises on every real divergence (changed cell, removed key, duplicate key, header drift) so the safety goal is preserved.
- Form-vs-graded comparison counts records, not unique students. The grader emits one output row per form submission, so record-vs-record is the matching comparison. If a future grader pass collapses multiple submissions per student, both `count_form_records` and `count_graded_records` move together to "unique Student IDs" and the dashboard threshold stays correct.
- `graded > form` does not flip status to PARTIAL, only emits a footer warning. PARTIAL routes to `regrade`, but a regrade cannot fix "more graded than submitted" -- that is a stale output (form CSV rolled back) and only the operator can decide whether to re-download the form, edit the output, or accept the discrepancy.
- Existing files saved under the legacy grader-only shape `<RUID>-<first>_<last>-<basename>.<ext>` are not auto-renamed by the unified-filename fix. Operator must remove duplicate copies from `<image_dir>/raw/`, `<image_dir>/trim/`, and the bank mirrors before re-running grade so only the canonical `<RUID>-protein<NN>-*` files remain. A one-shot rename helper is left as a follow-up. Same precedent as the RUID-policy fix below.
- Existing wrong-RUID files already on disk under `<image_dir>/raw/`, `<image_dir>/trim/`, and the bank mirrors are **not** renamed by the RUID-policy fix. Operator must clear semester `raw/`+`trim/` and re-run `download_submission_images.py` to regenerate canonical filenames. The bank still holds the typed-RUID copies; a one-shot rename helper is left as a follow-up.
- An earlier draft of this change introduced `protein_image_grader/ruid_aliases.py` to persist a `typed_ruid -> resolved_ruid` cache as `Protein_Images/semesters/<term>/ruid_aliases.yml`. Removed before merge. Reasoning: `roster.csv` is the only source of truth; a typed RUID that misses the roster is a one-off student typo, not durable data worth caching across runs.
